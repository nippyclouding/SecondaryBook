# Redis 사용 목적 및 필요성

> 본 프로젝트에서 Redis를 도입한 이유와 각 기능별 필요성을 정리한다.

---

## 아키텍처 개요

```
                        ┌─────────────────────┐
                        │   ALB (로드밸런서)     │
                        └──────┬──────┬───────┘
                               │      │
                    ┌──────────┘      └──────────┐
                    ▼                             ▼
             ┌─────────────┐              ┌─────────────┐
             │   EC2 #1    │              │   EC2 #2    │
             │  (Tomcat)   │              │  (Tomcat)   │
             └──────┬──────┘              └──────┬──────┘
                    │                             │
                    └──────────┐  ┌───────────────┘
                               ▼  ▼
                     ┌───────────────────┐
                     │  Redis(ElastiCache)│
                     └───────────────────┘
```

Auto Scaling 환경에서 여러 EC2 인스턴스가 동시에 운영되면, 각 인스턴스는 독립된 프로세스이므로 **서버 메모리를 공유할 수 없다**. Redis는 이 인스턴스들 사이에서 공유 저장소이자 메시지 브로커 역할을 한다.

---

## 1. Pub/Sub — 채팅 메시지 브로드캐스트

### 왜 필요한가

채팅 메시지는 DB에 영구 저장하지만, **실시간 전달**은 WebSocket(STOMP)이 담당한다. 문제는 WebSocket 세션이 서버 메모리에 존재한다는 것이다.

```
[Pub/Sub 없이 — EC2 2대 환경]

유저A(판매자) → EC2 #1에 WebSocket 연결
유저B(구매자) → EC2 #2에 WebSocket 연결

유저A가 메시지 전송
  → EC2 #1이 DB에 저장 ✅
  → EC2 #1의 SimpleBroker로 브로드캐스트
  → EC2 #1에 연결된 클라이언트에게만 전달
  → 유저B는 EC2 #2에 있으므로 실시간 수신 불가 ✗
```

DB에 저장했으니 유저B가 새로고침하면 볼 수는 있지만, **실시간 채팅**이 아니게 된다.

### Pub/Sub 적용 후

```
유저A가 메시지 전송
  → EC2 #1이 DB에 저장
  → EC2 #1이 Redis에 PUBLISH ("chat:messages" 채널)
  → Redis가 모든 구독 인스턴스에 메시지 전파
  → EC2 #1, EC2 #2 모두 SUBSCRIBE로 수신
  → 각 인스턴스가 자기에게 연결된 WebSocket 클라이언트에게 전달
  → 유저B도 실시간 수신 ✅
```

### 핵심 포인트

- DB 저장 = **영속성** (메시지를 잃지 않기 위함)
- Pub/Sub = **실시간 전달** (다른 서버의 WebSocket 클라이언트에게 즉시 전달)
- 이 둘은 **서로 다른 문제를 해결**하므로 DB에 저장한다고 Pub/Sub이 불필요해지지 않는다

### 대안과 비교

| 방식 | 장점 | 단점 |
|------|------|------|
| **Redis Pub/Sub** | 지연 거의 없음, 구현 간단 | Redis 장애 시 실시간 전달 불가 |
| DB 폴링 | Redis 불필요 | 폴링 주기만큼 지연, DB 부하 증가 |
| Sticky Session | Redis 불필요 | 특정 서버 장애 시 세션 유실, 부하 불균형 |

## 2. Session — 분산 세션 관리

### 왜 필요한가

Spring의 `HttpSession`은 기본적으로 서버 메모리(Tomcat)에 저장된다.

```
[Redis Session 없이]

유저가 EC2 #1에서 로그인 → 세션이 EC2 #1 메모리에 생성
  → 다음 요청이 ALB에 의해 EC2 #2로 라우팅됨
  → EC2 #2에는 해당 세션이 없음
  → 로그아웃 상태로 처리됨 ✗
```

### Redis Session 적용 후

```java
@EnableRedisHttpSession(
    maxInactiveIntervalInSeconds = 1800,  // 30분
    redisNamespace = "secondarybook:session"
)
```

모든 EC2 인스턴스가 Redis에서 세션을 읽고 쓰므로, 어떤 인스턴스로 라우팅되든 동일한 세션을 사용한다.

```
유저가 EC2 #1에서 로그인 → 세션이 Redis에 저장
  → 다음 요청이 EC2 #2로 라우팅됨
  → EC2 #2가 Redis에서 세션을 조회
  → 로그인 상태 유지 ✅
```

### 대안과 비교

| 방식 | 장점 | 단점 |
|------|------|------|
| **Redis Session** | 서버 무상태, 스케일링 자유 | Redis 의존성 추가 |
| Sticky Session (ALB) | 구현 간단 | 서버 장애 시 세션 유실, 부하 불균형 |
| DB에 세션 저장 | Redis 불필요 | 매 요청마다 DB 조회, 성능 저하 |
| JWT (토큰 기반) | 서버 무상태, Redis 불필요 | 토큰 탈취 시 만료까지 무효화 불가, 구현 복잡 |

### Redis Session을 선택한 이유

- 기존 `HttpSession` 기반 코드를 **거의 수정하지 않고** 분산 환경을 지원할 수 있다
- `@EnableRedisHttpSession` 어노테이션 하나로 적용 가능
- WebSocket 핸드셰이크 시 세션 검증에도 동일하게 동작한다

## 3. Cache — 조회 성능 최적화

### 왜 필요한가

중고 거래 목록, 독서모임 목록처럼 **자주 조회되지만 변경은 드문 데이터**는 매번 DB를 조회할 필요가 없다.

### 캐시 적용 현황

| 캐시 이름 | TTL | 대상 | 사용 위치 |
|-----------|-----|------|----------|
| `tradeList` | 5분 | 중고거래 목록 | `TradeService` |
| `trade` | 10분 | 중고거래 상세 | `TradeService` |
| `tradeCount` | 10분 | 중고거래 개수 | `TradeService` |
| `bookClubList` | 10분 | 독서모임 목록 | `BookClubService` |
| `bookClub` | 10분 | 독서모임 상세 | `BookClubService` |

### 동작 방식

```java
// 첫 번째 조회: DB에서 가져온 후 Redis에 캐싱
@Cacheable(value = "trade", key = "#trade_seq")
public TradeVO search(long trade_seq) { ... }

// 데이터 변경 시: 캐시 무효화
@CacheEvict(value = "tradeList", allEntries = true)
public void update(TradeVO tradeVO) { ... }
```

```
1차 요청: DB 조회 → Redis에 저장 → 응답 (느림)
2차 요청: Redis에서 조회 → 응답 (빠름)
데이터 변경: 캐시 삭제 → 다음 조회 시 DB에서 다시 가져옴
```

### 이것은 오토스케일링과 무관하게 유용하다

EC2가 1대여도 캐시는 DB 부하를 줄여준다. 다만 **여러 EC2가 Redis 캐시를 공유**하므로, 한 인스턴스에서 캐시를 적재하면 다른 인스턴스도 즉시 활용할 수 있다는 부가 이점이 있다.

### TTL을 짧게 설정한 이유

- 중고거래 특성상 **가격 변경, 판매 완료** 등이 빈번하다
- 오래된 캐시로 인해 이미 팔린 상품이 목록에 남아있으면 사용자 경험이 나빠진다
- 5~10분의 짧은 TTL + 변경 시 `@CacheEvict`로 즉시 무효화하여 **정합성과 성능의 균형**을 맞췄다

## 4. 이메일 인증코드 — 임시 데이터 저장

### 왜 필요한가

회원가입 시 이메일 인증코드를 발급하고, 사용자가 입력한 코드와 대조해야 한다. 이 코드는 **3분 후 자동 만료**되어야 한다.

```java
// 인증코드 저장 (3분 TTL)
redisTemplate.opsForValue().set(
    "AuthCode:" + email, checkNum, 180, TimeUnit.SECONDS
);

// 인증코드 검증
String storedCode = redisTemplate.opsForValue().get("AuthCode:" + email);
```

### Redis를 사용하는 이유

| 방식 | 문제점 |
|------|--------|
| 서버 메모리 (Map) | EC2 #1에서 발급한 코드를 EC2 #2에서 검증 불가 |
| DB 저장 | 가능하지만 만료 처리를 별도 스케줄러로 구현해야 함 |
| **Redis** | TTL 자동 만료 지원, 다중 서버에서 공유 가능 |

### 핵심 포인트

- Redis의 `TTL(Time To Live)` 기능으로 **만료 로직을 별도 구현할 필요 없이** 자동 삭제된다
- 영속성이 필요 없는 임시 데이터이므로 DB보다 Redis가 적합하다

## 정리

### "프로젝트에서 Redis를 사용한 이유"

> Auto Scaling으로 EC2가 여러 대 운영되는 환경에서, 각 인스턴스가 서버 메모리를 공유할 수 없는 문제를 해결하기 위해 Redis를 도입, 크게 4가지 용도로 사용한다.

### "각각의 용도"

| 용도 | 한 줄 요약 |
|------|-----------|
| **Pub/Sub** | 채팅 메시지를 모든 서버 인스턴스의 WebSocket 클라이언트에게 실시간 전달 |
| **Session** | 어떤 EC2로 요청이 라우팅되든 동일한 로그인 세션 유지 |
| **Cache** | 자주 조회되는 목록/상세 데이터를 캐싱하여 DB 부하 감소 |
| **인증코드** | TTL 기반 자동 만료로 이메일 인증코드의 임시 저장 |

### "DB에 저장하는데 Pub/Sub이 필요한 이유?"

> DB 저장은 메시지의 영속성을 보장하고, Pub/Sub은 실시간 전달을 담당, DB에 저장했더라도 다른 EC2에 연결된 WebSocket 클라이언트에게 즉시 전달할 방법이 없으므로, Redis Pub/Sub을 통해 모든 인스턴스에 메시지를 브로드캐스트한다. 이 둘은 서로 다른 문제를 해결한다(다중 서버에서 DB, Redis 모두 필요)

### "EC2가 1대면 Redis가 필요 없지 않는지"

> Pub/Sub과 Session은 다중 인스턴스 환경에서만 필요하다. 하지만 Cache와 인증코드 저장은 단일 서버에서도 유용하다. 또한 Auto Scaling을 사용하는 이상 언제든 인스턴스가 추가될 수 있으므로, 처음부터 Redis 기반으로 설계하는 것이 올바른 접근이라고 생각한다.
